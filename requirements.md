# Banana Player 项目方案书（MacOS 本地音乐播放器）

## 1. 文档目标
本文档用于定义 Banana Player 的首期方案，作为后续由 Claude 执行开发工作的统一依据。目标是：

- 在 macOS 平台实现轻量、本地、低硬件要求的音乐播放器。
- 首期聚焦「识别文件夹中的音乐并播放」这一核心闭环。
- 首期支持专辑维度浏览，能够识别专辑文件夹、专辑封面和乐曲音轨号。
- 在架构上确保模块边界清晰、可扩展、便于后续功能迭代。
- UI 风格遵循 Apple 平台设计规范（Human Interface Guidelines, HIG）。

## 2. 项目定位与原则

### 2.1 项目定位
- 产品类型：本地离线音乐播放器（非流媒体）。
- 目标用户：macOS 用户，偏好本地音乐管理与播放。
- 首期价值：快速导入本地音乐目录，稳定播放，界面简洁自然。

### 2.2 核心原则
- 轻量优先：避免不必要依赖，降低内存和 CPU 占用。
- 本地优先：不依赖网络即可完成核心功能。
- 稳定优先：播放稳定性高于复杂功能堆叠。
- 扩展优先：架构先行，为播放列表、歌词、均衡器、插件化预留空间。

## 3. 版本范围定义

### 3.1 MVP（首期必须完成）
- 选择并授权本地音乐文件夹。
- 扫描并识别常见音频文件。
- 识别专辑文件夹并按专辑聚合歌曲。
- 识别专辑封面（嵌入式封面或文件夹封面图片）。
- 识别乐曲音轨号并在专辑内按音轨顺序展示。
- 展示专辑列表与歌曲列表（至少包含文件名、时长、路径信息）。
- 支持基础播放控制：播放、暂停、上一首、下一首、拖动进度、音量调节。
- 支持连续播放当前列表。
- 支持流程：打开软件 -> 点击专辑封面进入专辑歌曲列表 -> 点击歌曲进行播放。
- 基础异常处理：无权限、无可播放文件、文件损坏、格式不支持。

### 3.2 非 MVP（明确不在首期）
- 在线音乐、账号系统、云同步。
- 歌词下载、在线封面抓取、推荐算法。
- 高级均衡器、音频效果器、可视化频谱。
- 多设备同步播放。

## 4. 技术选型建议（首期）

### 4.1 开发语言与框架
- 语言：Swift（建议 Swift 6 兼容写法）。
- UI：SwiftUI（必要处可桥接 AppKit）。
- 音频播放：AVFoundation（AVAudioEngine/AVPlayer 视实现选择）。
- 文件访问与扫描：FileManager + URLResourceValues。
- 本地持久化：首期可用轻量缓存（JSON/SQLite 二选一，建议 SQLite）。

### 4.2 选型理由
- 与 Apple 生态一致，长期维护成本低。
- 原生音频与 UI 能力完善，依赖体积可控。
- 对低硬件设备更友好，性能可预测。

## 5. 系统架构设计

### 5.1 架构风格
建议采用「分层 + MVVM + 面向协议」：

- Presentation 层：SwiftUI + ViewModel。
- Domain 层：用例（UseCase）与实体模型（Entity）。
- Data 层：文件扫描、元数据读取、播放引擎适配、仓储实现（Repository）。

### 5.2 模块划分
- AppCore：应用生命周期、依赖注入、全局配置。
- LibraryScanner：目录授权、递归扫描、增量更新。
- MetadataService：读取标题、艺术家、专辑、时长等标签。
- AlbumOrganizer：专辑聚合、封面选择策略、音轨排序策略。
- PlaybackEngine：播放队列、状态机、播放控制。
- UIHome：专辑封面墙、专辑歌曲列表、播放器控制条。
- Persistence：索引缓存、设置项保存。

### 5.3 模块边界约束
- View 不直接访问文件系统与播放器底层。
- 通过协议隔离实现：例如 MusicRepositoryProtocol、PlaybackControlling。
- 模块之间仅通过 Domain Model 交互，避免传递底层框架对象。

## 6. 核心业务流程

### 6.1 首次导入流程
1. 用户点击「添加音乐文件夹」。
2. 系统弹出 macOS 文件夹选择器并授权。
3. 记录安全访问书签（security-scoped bookmark，若启用沙盒）。
4. 执行扫描与元数据解析。
5. 刷新列表并允许播放。

### 6.2 扫描与索引流程
1. 读取目录下候选文件（支持递归开关，首期默认递归）。
2. 依据扩展名和可解码性过滤。
3. 读取元数据并写入索引缓存。
4. 按专辑字段与目录结构进行专辑聚合。
5. 识别专辑封面（优先嵌入封面，其次目录内 cover/folder 图片）。
6. 按音轨号（track number）排序，缺失时回退文件名排序。
7. 对重复文件（路径或文件哈希）去重。
8. 生成专辑列表与可播放列表。

### 6.3 专辑入口播放流程（关键）
1. 用户打开应用，默认进入专辑封面列表页。
2. 用户点击某个专辑封面，进入该专辑歌曲列表页。
3. 用户点击目标歌曲。
4. PlaybackEngine 加载音频资源。
5. 状态机切换到 buffering/playing。
6. 周期性回传播放进度给 ViewModel。
7. 播放结束自动切到下一首（若有）。

## 7. 功能需求清单（MVP）

### 7.1 文件夹管理
- FR-001：支持添加一个或多个本地音乐文件夹。
- FR-002：支持移除已添加文件夹。
- FR-003：应用重启后保留已添加文件夹信息。

### 7.2 音乐扫描与识别
- FR-010：支持识别常见格式：mp3、m4a、aac、wav、aiff（FLAC 可作为可选支持）。
- FR-011：显示扫描进度与结果统计（总文件数、可播放数、失败数）。
- FR-012：对不可读文件给出可理解错误提示。
- FR-013：支持识别专辑信息并将歌曲按专辑归类。
- FR-014：支持识别专辑封面，优先级为：内嵌封面 > folder.jpg/png > cover.jpg/png。
- FR-015：支持识别乐曲音轨号并用于专辑内默认排序。

### 7.3 播放控制
- FR-020：播放/暂停。
- FR-021：上一首/下一首。
- FR-022：拖动进度条跳转。
- FR-023：音量控制与静音。
- FR-024：单曲结束自动播放下一首。

### 7.4 专辑与交互
- FR-030：主页展示专辑封面网格（至少包含封面、专辑名、艺术家）。
- FR-031：点击专辑封面后进入该专辑歌曲列表。
- FR-032：专辑歌曲列表按音轨号升序展示，缺失时回退到文件名排序。
- FR-033：点击歌曲后立即开始播放，并同步高亮当前播放曲目。
- FR-034：支持关键词筛选（专辑名、歌曲名）。

## 8. 非功能需求

### 8.1 性能指标（目标值）
- NFR-001：冷启动时间 <= 1.5s（中等规模库，约 3000 首）。
- NFR-002：空闲状态 CPU 占用 < 3%。
- NFR-003：播放状态 CPU 占用（无特效）< 8%。
- NFR-004：空闲内存占用 < 150MB。
- NFR-005：首次点击播放至出声延迟 < 300ms（本地文件）。

### 8.2 兼容性
- NFR-010：系统版本建议 macOS 13+。
- NFR-011：Apple Silicon 与 Intel 均可运行。

### 8.3 稳定性
- NFR-020：连续播放 2 小时无崩溃。
- NFR-021：损坏音频文件不导致应用崩溃。

## 9. UI/UX 设计要求（Apple 规范导向）

### 9.1 总体风格
- 使用 macOS 原生布局语义：Sidebar + Content + Bottom/Inline Controls。
- 优先使用系统控件与系统字体层级（SF Pro Text / Display）。
- 图标优先使用 SF Symbols，确保平台一致性。
- 首页信息架构优先按「专辑」组织，减少纯文件列表直出造成的杂乱感。

### 9.2 交互规范
- 支持常用快捷键：空格（播放/暂停）、Command+O（添加文件夹）。
- 支持窗口尺寸变化下的自适应布局。
- 提供明确的空状态与错误状态视图。
- 专辑封面可点击进入详情，歌曲行点击后直接播放。

### 9.3 无障碍
- 控件具备 VoiceOver 可读标签。
- 关键文本和控件满足对比度要求。

## 10. 数据与模型设计（建议）

### 10.1 核心实体
- Album
  - id: String
  - title: String
  - artist: String?
  - coverURL: URL?
  - trackCount: Int
- Track
  - id: String
  - fileURL: URL
  - title: String
  - artist: String?
  - album: String?
  - albumID: String?
  - trackNumber: Int?
  - discNumber: Int?
  - artworkURL: URL?
  - duration: Double
  - format: String
- LibraryFolder
  - id: String
  - folderURL: URL
  - bookmarkData: Data?
- PlaybackState
  - currentTrackID: String?
  - isPlaying: Bool
  - progress: Double
  - volume: Float

### 10.2 持久化策略
- 首期建议：
  - 配置类数据（文件夹、用户偏好）存储在 UserDefaults + 文件。
  - 曲库索引存储在 SQLite（便于后续查询扩展）。

## 11. 可扩展性设计要求

### 11.1 扩展点预留
- PlaybackControlling：可替换播放引擎（后续支持高级音频处理）。
- MetadataProviding：可替换标签读取实现（后续支持更多格式）。
- QueueStrategy：可注入顺序播放、随机播放、单曲循环策略。

### 11.2 演进路线（建议）
- Phase 2：播放列表、最近播放、收藏。
- Phase 3：歌词与封面管理。
- Phase 4：插件机制与 DSP 扩展。

## 13. 开发里程碑

### M1：基础框架搭建（1 周）

- 打通基础 UI 壳与播放器核心接口。

### M2：曲库扫描与列表（1 周）

## 14. 验收标准（MVP）
- 能成功导入指定本地文件夹并识别可播放文件。
- 能识别专辑文件夹并展示专辑封面列表。
- 能从专辑封面进入专辑歌曲列表，并按音轨号排序（缺失时合理回退）。
- 能点击专辑中的任意歌曲并立即播放。
- 能稳定播放、暂停、切歌、拖动进度。
- 重启应用后仍保留曲库与基本设置。
- 对异常文件或无权限目录有明确提示，不崩溃。
- 在目标硬件上达到第 8 章定义的核心性能指标。
